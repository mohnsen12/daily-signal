The $0 Tech Stack Behind an AI-Written Newsletter

18. marts 2026 · 10 min læsetid · Architecture breakdown for developers

Every morning at 6 AM, an AI agent researches 50+ sources, writes a newsletter, checks quality, and publishes — all for $0/month. No SaaS subscriptions. No API costs. Just open-source tools on a Mac mini that was already running.

Here's the complete architecture.

The Pipeline

The entire system is a chain of cron-triggered agent sessions. Each step is isolated and can fail independently:

⏰ Cron 06:00 → 🔍 Research → ✍️ Write → ✅ Quality Check → 📄 Generate HTML → 🃏 Generate Card → 📦 Archive + RSS → 📊 Analytics → 🚀 Deploy

Step 1: Research (5 min)

The agent uses web search + web fetch to scan 50+ sources. Each source is scored on relevance, novelty, and signal-to-noise. Stories are deduplicated (same event, different outlet = merge).

// Simplified research prompt structure
const researchPrompt = `
  Scan these sources: ${sources.join(', ')}
  For each story, rate:
  - Relevance: 1-5 (how important for tech/AI audience?)
  - Novelty: 1-5 (is this new information?)
  - Signal: 1-5 (will readers care?)
  Pick top 6-8 by combined score.
  Output: JSON array of {headline, summary, source, url, scores}
`;

Step 2: Write (15 min)

The agent follows a structured template (v3) with 10 sections:

  1. TL;DR — 5 bullet points for busy readers
  2. Breaking — Top 2 stories with deep analysis
  3. Products — 3 launches with pricing/audience
  4. Analysis — 2 stories with "why it matters" + action items
  5. Danish Angle — What it means for Denmark
  6. Tool of the Day — One AI tool with honest review
  7. Stats — Key numbers in a visual table
  8. Teddy's Take — The AI's personal opinion (unique)
  9. Reader Question — Open question to drive engagement
  10. Shareable Quote — Pre-formatted for social sharing

The template enforces structure. The AI fills in the content. This combination produces consistency that pure "write me a newsletter" prompting can't match.

Step 3: Quality Check (2 min)

A separate agent session scores the newsletter on 15 criteria:

const qualityChecks = [
  { name: 'TL;DR present', points: 10 },
  { name: 'Tool of the Day', points: 10 },
  { name: 'Stats section', points: 10 },
  { name: 'Analysis with "why"', points: 10 },
  { name: 'Danish angle', points: 10 },
  { name: 'Teddy\'s take', points: 10 },
  { name: 'Reader question', points: 10 },
  { name: 'Action items', points: 5 },
  { name: 'Min 5 stories', points: 10 },
  { name: 'All links valid', points: 5 },
  { name: 'Shareable quote', points: 5 },
  { name: 'About section', points: 5 },
  // ... total: 115 points possible
];

Score below 90? The agent rewrites the missing sections and re-checks. This feedback loop is the key to quality improvement.

Pro tip: The quality check runs as a separate agent session with a different prompt. This means it's an independent "editor" — not the same AI grading its own homework.

Step 4-9: Generate Everything

After quality check passes, the system generates:

  • HTML email — Responsive dark-mode email with inline CSS
  • Shareable card — Standalone HTML page for social sharing
  • Archive — Landing page updates with new issue
  • RSS — Feed for RSS readers
  • Sitemap — SEO-optimized sitemap.xml
  • Tweet — Auto-generated tweet from top story
  • Analytics — Quality score, word count, story count logged

All deployed to GitHub Pages automatically.

Key Architecture Decisions

1. Cron jobs, not daemon processes

Each step is an isolated agent session triggered by cron. This means:

  • No memory leaks (sessions are ephemeral)
  • No "stuck" processes (each run completes or times out)
  • Easy to debug (each run has its own history)
  • Easy to modify (change the prompt, next run uses it)

2. Quality feedback loop

The most important innovation is the write → check → rewrite loop. The quality check agent is a separate "editor" that scores independently. If the score drops below 90, the writer agent rewrites. This creates a ratchet effect where quality only goes up.

3. Template-driven content

The template is the secret sauce. Without it, AI-generated newsletters are inconsistent. With it, every issue has the same structure, making it predictable for readers and improvable over time.

Honest limitation: This system can't build genuine community. It can write great content, but it can't reply to reader emails with warmth, remember personal details, or build the kind of relationship that makes people recommend newsletters to friends. That still requires a human.

What I'd Do Differently

  1. Start with email service from day one. localStorage signups are useless. Use Buttondown (free for 100 subs) from the start.
  2. Don't build promo material before having readers. I have 7+ promo files ready. They're sitting unused because there's no audience yet.
  3. Deploy distribution early. Content without distribution is a diary. Auto-tweets, Reddit posts, and social sharing should be part of the initial pipeline, not afterthoughts.
  4. Track from day one. Analytics should be built into the first version, not added on day 3.

Can You Build This?

Yes. The entire system is built on open-source tools:

  • Agent framework: OpenClaw (handles cron, memory, tool use)
  • AI models: Any open-source LLM via OpenRouter or local
  • Hosting: GitHub Pages (free, static)
  • Version control: Git

The hardest part isn't the tech — it's the editorial judgment baked into the prompts and templates. That's the "secret sauce" that took 5 days of iteration to get right.
